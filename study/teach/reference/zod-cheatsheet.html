Zod v4 cheat sheet

The validators you'll reach for building Applyo's forms — mapped to real fields. Zod v4 (import * as z from "zod").

v3 → v4 Format checks moved to the top level: it's z.email(), z.url(), z.uuid() — not z.string().email(). And error messages use one error parameter (a string or function), replacing v3's message / required_error / invalid_type_error.

Strings — title, company, notes

z.string()                      // any string
z.string().min(1, "Title is required")   // non-empty
z.string().max(120)
z.string().trim()               // transforms: strips whitespace

Formats — sourceUrl, a contact email

z.url()                          // valid URL string
z.url({ protocol: /^https?$/ }) // http/https only
z.email("Enter a valid email")   // error msg as 1st arg

Enums — opportunity status, pipeline stage

z.enum(["saved", "reviewing", "ready_to_apply",
        "ignored", "expired", "converted"])
// input must be exactly one of these strings

Numbers & coercion — salary, match score

Why coerce Every value from an HTML <input> arrives as a string — even a number field. z.coerce.number() runs Number(input) first, so "5000" becomes 5000.
z.number().int().gte(0).lte(100)   // match score 0–100
z.coerce.number()                    // "5000" → 5000
z.coerce.date()                      // "2026-07-12" → Date

Cross-field rules — salary min ≤ max

z.object({
  salaryMin: z.coerce.number().optional(),
  salaryMax: z.coerce.number().optional(),
}).refine(
  (d) => !d.salaryMin || !d.salaryMax || d.salaryMin <= d.salaryMax,
  { error: "Min salary can't exceed max", path: ["salaryMax"] }
)
path The path tells the UI which field to show the error under. Without it the issue attaches to the whole object.

Massaging input — coerce vs preprocess vs transform vs pipe

One question tells them apart Does it run before the validation gate (fixing input so it can pass) or after (reshaping a value already valid)? See Lesson 2.
Not on this list: .refine() These four all massage (change) the value. .refine() is a validation check — it inspects the value and passes it through unchanged or raises an issue. Mirror of .transform(): transform reshapes but can't reject; refine can reject but doesn't reshape. Covered in Lesson 4.
ToolRunsUse when
z.coerce.number()before gatebuilt-in JS conversion is exactly right ("5000"→5000)
z.preprocess(fn, S)before gateyou need a custom rule on raw input (blank → undefined)
.transform(fn)after gatereshape/derive a value you know is valid; may change output type
A.pipe(B)gate → gateeach step is itself a schema (string, then ISO date)
The coerce trap z.coerce.number().optional() turns "" and null into 0 — .optional() only catches undefined. For "blank means absent," normalise with preprocess first.
z.preprocess(v => v === "" ? undefined : Number(v),
             z.number().nonnegative().optional())
// input type: z.input<typeof S>   output type: z.infer<typeof S> — they differ once you massage

Reading errors in a form — see Lesson 3

Every issue = { code, path, message }. path is the field's address: ["salaryRange","max"], ["requiredTechnologies", 1]. Join it for a key.

ToolReturnsUse for
r.error.issuesflat [{ path, message, code }]full control — map it yourself
z.flattenError(e){ formErrors, fieldErrors }flat forms; collapses nested/array paths to top key
z.treeifyError(e)nested .properties/.items/.errorsnested objects & arrays (salaryRange, tech list)
z.prettifyError(e)human-readable stringlogs / console, not per-field UI
const r = OpportunitySchema.safeParse(formData);
if (!r.success) {
  // map issues → one message per field
  const errs = {};
  for (const i of r.error.issues)
    errs[i.path.join(".")] ??= i.message;
}
gotchas A field can have several issues (show the first). path: [] = a whole-form error → flattenError.formErrors. z.formatError is deprecated → use treeifyError.
