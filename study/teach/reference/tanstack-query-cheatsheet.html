TanStack Query v5 cheat sheet

The query mechanics you'll reach for wiring Applyo's data layer — mapped to real endpoints. React v5 (@tanstack/react-query). Grows as the course does.

v4 → v5 Single object signature only: useQuery({ queryKey, queryFn }) (no positional args). "No data yet" is isPending, not isLoading (isLoading now = isPending && isFetching, i.e. the first load). Cache GC time is gcTime, not cacheTime.

TanStack Query is an async state manager, not a fetch wrapper: it caches a copy of server-owned data and decides when to refetch.

The anatomy

const { data, isPending, isError, error, isFetching } = useQuery({
  queryKey: ["opportunities"],   // cache ADDRESS — same key ⇒ same data, shared request
  queryFn: fetchOpportunities,    // returns a Promise; MUST throw on failure
});

queryFn — the throw rule

async function fetchOpportunities(): Promise<Opportunity[]> {
  const res = await fetch("/api/opportunities");
  if (!res.ok) throw new Error("Failed to load opportunities");
  return res.json();          // fetch does NOT reject on 4xx/5xx — you must throw
}

Fresh / stale & the two clocks

Every cached query is fresh (served from cache, no request) or stale (still shown, but refetch-eligible). staleTime is the age at which fresh → stale.

staleTimegcTime (was cacheTime)
GovernsTrust — when to refetchRetention — when to delete
Default0 — stale immediately5 min
Clock startswhen data arriveswhen query goes unused (no observer)
On expiryshown, but refetch-eligibledeleted from cache
useQuery({ queryKey, queryFn, staleTime: 20_000 });  // per-query
new QueryClient({ defaultOptions: { queries: { staleTime: 30_000 } } }); // global

A stale query auto-refetches on: mount (refetchOnMount) · window refocus (refetchOnWindowFocus) · reconnect (refetchOnReconnect). With staleTime: 0 that's every such event; raise it and those events no-op while fresh. On a background refetch old data stays on screen: isPending false, isFetching true. Trap: staleTime ≠ retention — staleTime: Infinity data is still GC'd 5 min after nothing renders it.

queryKey patterns

["opportunities"]                          // the whole list
["opportunities", { status: "saved" }]      // a filtered list — different address
["opportunity", id]                        // one item by id

Keys are arrays, compared by value (deep). Put every variable the queryFn depends on in the key — change the inputs, change the address, get a fresh cache entry (and an automatic refetch).

App Router setup (Next.js 16)

// src/app/providers.tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export default function Providers({ children }: { children: React.ReactNode }) {
  const [client] = useState(() => new QueryClient());  // once per client, not module top-level
  return <QueryClientProvider client={client}>{children}</QueryClientProvider>;
}
// then wrap {children} with <Providers> in src/app/layout.tsx
Two rules that cause 90% of "it doesn't work" 1. useQuery must run inside a Client Component ("use client") under the provider. 2. Never create the QueryClient at module top-level — on the server one instance would be shared across all requests/users. useState(() => new QueryClient()) pins one per client.
