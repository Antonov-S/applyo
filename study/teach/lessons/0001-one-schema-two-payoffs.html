Applyo · Zod course · Lesson 1

One schema, two payoffs

Write the Opportunity form's rules once. Get a runtime validator and its TypeScript type out the other side — no hand-written interface to keep in sync.

You've written a Zod schema before, so z.object({...}) won't be new. This lesson is about the two things that make Zod worth reaching for — and both land right on the spot where TypeScript still feels fuzzy.

The problem Zod solves

Without Zod, saving an opportunity means writing the same knowledge twice:

// 1. A type — so the rest of the app knows the shape
interface Opportunity {
  title: string;
  company: string;
  sourceUrl?: string;
}

// 2. Runtime checks — because the type vanishes at runtime
if (typeof input.title !== "string" || input.title === "") { /* ... */ }
if (typeof input.company !== "string") { /* ... */ }
// ...and these two drift apart the moment you add a field
The key idea A TypeScript type is erased before your code runs — it can't check the JSON a form actually submits. So you need runtime checks too. Zod flips this: you write the runtime schema, and TypeScript reads the type off it. One source of truth, both jobs done.

Payoff 1 — the schema validates at runtime

Here's the same three fields as a schema, and the way you'll actually call it in a form handler — safeParse:

import * as z from "zod";

const OpportunitySchema = z.object({
  title:     z.string().min(1, "Title is required"),
  company:   z.string().min(1, "Company is required"),
  sourceUrl: z.url().optional(),
});

const result = OpportunitySchema.safeParse(formInput);

safeParse never throws. It hands back one of two shapes — a discriminated union, which is just a fancy way of saying "check one flag and TypeScript knows which shape you have":

if (result.success) {
  result.data;    // ✅ clean, typed Opportunity — safe to save
} else {
  result.error;   // ❌ a ZodError; result.error.issues lists what failed
}

Use safeParse (not parse) for form input: bad input from a user is expected, not an exception. parse throws on failure — save it for places where invalid data is a genuine bug.

Payoff 2 — the type comes for free

This is the line that removes work from your plate. Instead of hand-writing interface Opportunity, you ask the schema for its type:

type Opportunity = z.infer<typeof OpportunitySchema>;
// ⇒ { title: string; company: string; sourceUrl?: string }
Reading that line slowly typeof OpportunitySchema = "the type of the schema value." z.infer<...> = "now dig the data type out of it." You never write the field list twice. Add closingDate to the schema and the Opportunity type grows to match — automatically. The two-can't-drift-apart property is the whole point.

And because result.data already is that type, this just works with no casting:

if (result.success) {
  saveOpportunity(result.data);  // result.data : Opportunity
}

2. You add notes: z.string().optional() to the schema. What must you change about the Opportunity type?

Your win for today

In your Applyo repo, create a schema file and prove the loop to yourself:

  1. Write OpportunitySchema with title, company, and sourceUrl (start with just these three).
  2. Export type Opportunity = z.infer<typeof OpportunitySchema>.
  3. Call safeParse on a good object and a bad one; console.log both results and read the issues array on the failure.
  4. Hover the Opportunity type in your editor — confirm it matches the schema with nothing hand-written.

Suggested home: src/lib/schemas/opportunity.ts. You don't need a form or a component yet — a scratch file and the terminal are enough to see it work. (This is a spike: no tests required, just the learning note.)
