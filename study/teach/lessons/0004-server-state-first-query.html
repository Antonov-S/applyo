Server state, and your first query

TanStack Query is not a nicer fetch(). It's a cache that holds a copy of data the server owns, and tells you when your copy has gone stale. Wire one real query against your opportunities and you get loading, errors, caching, request dedup, and background refresh — without writing any of it.

You've written a useQuery before, so this lesson skips "what's a hook." It spends its budget on the two ideas that separate copying a useQuery from understanding one: why server data is a different kind of state, and why the queryKey is the whole trick.

Why not useState + useEffect?

Here's the opportunities list fetched by hand — the version you'd otherwise write:

const [data, setData] = useState<Opportunity[]>([]);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  fetch("/api/opportunities")
    .then(r => r.json())
    .then(setData)
    .catch(setError)
    .finally(() => setLoading(false));
}, []); // runs once per mount — per component, every time

It looks fine. What's missing is everything that makes it correct: two components that both need opportunities fire two requests (no dedup); the data is thrown away and refetched on every mount (no cache); a fast-then-slow response can overwrite fresh data with stale (a race); and nothing ever refreshes it after the first load (no background update). You can hand-code all four. TanStack Query is the library that already did.

The shift: server state ≠ client state

This is the mental model the whole library hangs on. There are two kinds of state, and you've been treating them the same:

Client stateYou own it. A modal's open flag, the filter you typed, form input. It's true the instant you set it.
Server stateYou borrow it. The opportunities row in Postgres. You hold a copy; the real thing lives elsewhere and can change without telling you.

Client state (useState, Zustand, Redux) is happy because you're the source of truth. Server state is "totally different" (TanStack docs): it lives remotely, changes without your knowledge, and goes stale if you neglect it. That's why the maintainer calls TanStack Query not a data-fetching library but an "async state manager" (TkDodo) — its job is keeping your borrowed copy usefully in sync.

The reframe in one line You're not fetching your opportunities — you're caching a copy of server-owned data and letting the library decide when the copy is stale enough to refetch. Fetching is an implementation detail of that job.

Anatomy of a query

The v5 hook takes a single object. Two fields carry the whole idea:

const { data, isPending, isError, error } = useQuery({
  queryKey: ["opportunities"],          // the cache ADDRESS
  queryFn: fetchOpportunities,           // returns a Promise, THROWS on failure
});

queryKey — the cache address. This is the trick, not decoration. The key uniquely identifies this data in the cache. Call useQuery with the same key anywhere in the tree and you get the same cached data, one shared request — that's where dedup, caching, and "refetch this exact thing" all come from (Query Keys). Later, ["opportunities", { status: "saved" }] becomes a different address — a different cache entry — which is how filtered lists will just work.

queryFn — returns a Promise, and must throw on failure. Here's the gotcha that bites everyone: fetch() does not reject on a 404 or 500. If your queryFn doesn't check res.ok and throw, TanStack Query thinks a failed request succeeded and you never reach the error state:

async function fetchOpportunities(): Promise<Opportunity[]> {
  const res = await fetch("/api/opportunities");
  if (!res.ok) throw new Error("Failed to load opportunities"); // ← without this, no error state
  return res.json();
}

2. What is the queryKey primarily for?

Reading the result: two questions, not one

A query answers two independent questions, and mixing them up is the classic "used it lightly" bug. Straight from the docs:

FieldQuestion it answersValues
statusDo we have data yet?pending · error · success
fetchStatusIs the queryFn running?fetching · paused · idle

The booleans you'll actually branch on: isPending (status === "pending" — no data yet, show a skeleton), isError (show error.message), isSuccess (render data). Separately, isFetching is true whenever the queryFn runs — including a background refetch while old data is still on screen. That split is the point: on a background refresh you keep showing cached opportunities (isPending is false, you have data) while isFetching quietly powers a little spinner.

v5 naming — so old tutorials don't trip you In v5 the "no data yet" flag is isPending (v4 called it isLoading). isLoading still exists but now means isPending && isFetching — the first load specifically. Reach for isPending for "do I have data?" and you'll be right.

Your win for today

A real query in Applyo, reading real (mock) opportunities. The provider is boilerplate (infra — take it as given); the query and the states are yours to write. I review after.

Given to you — the provider (App Router gotcha baked in)

The QueryClient must be created once per client, not at module top-level (on the server that one instance would leak cache across requests/users). useState pins it. The provider is a Client Component:

// src/app/providers.tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export default function Providers({ children }: { children: React.ReactNode }) {
  const [client] = useState(() => new QueryClient());
  return <QueryClientProvider client={client}>{children}</QueryClientProvider>;
}

Then wrap {children} with <Providers> in src/app/layout.tsx. useQuery only works inside a Client Component under this provider.

Yours to build

  1. npm i @tanstack/react-query. (Optional but recommended: @tanstack/react-query-devtools — a panel that shows the cache live.)
  2. Drop in the provider above and wrap the layout.
  3. Write the Route Handler src/app/api/opportunities/route.ts — a GET returning NextResponse.json(MOCK), where MOCK: Opportunity[] is 2–3 hand-written opportunities. Type the array with your own Opportunity from @/lib/schemas/opportunity — TypeScript will make you fill every required field, which is the point.
  4. Write fetchOpportunities (with the res.ok throw) and a Client Component that calls useQuery and renders all three states: pending → "Loading…", error → error.message, success → the list of titles.
  5. Prove the shared cache — in one tab: render your list component twice on the same page (same queryKey: ["opportunities"]). Watch the Network panel — you'll see one request, not two. Both components read the same cache entry; the queryKey is acting as a shared address. Why not two tabs? Each tab is a separate document with its own new QueryClient() and its own in-memory cache — nothing is shared across tabs. The cache lives in one JS runtime.

Want the "second mount serves from cache" version instead? Add a second route, put a <Link> between them, and navigate away and back in the same tab — the QueryClient survives client navigation, so no refetch fires within the default stale window. (That window is Lesson 2.)
