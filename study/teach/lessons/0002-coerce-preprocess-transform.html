Coerce, preprocess, transform

You already reached for z.preprocess in your salary schema — and, it turns out, for exactly the right reason. This lesson makes that instinct explicit: three tools that reshape data, told apart by one question — where do they run relative to validation?

This is your question from last session — "when do I use z.preprocess vs z.coerce vs .transform()?" It's the right seam to press on, because all three look interchangeable and are not. Once you see the pipeline, the choice stops being a guess.

The mental model: a schema is a pipeline with a gate

Every Zod schema runs raw input through a validation gate. Some tools massage the value before the gate; one reshapes it after. That's the whole distinction.

The one-line test Ask: "Is my value valid yet?" If you're fixing input so it can pass → that's a before tool (coerce or preprocess). If you're reshaping something that already passed → that's .transform(), after.

coerce vs preprocess — the empty-string trap (your salary field)

Here's the choice you actually made. Both fill a number from a form field. Watch what each does with an empty salary input — the common case, since most vacancies list no salary:

// The tempting one-liner
z.coerce.number().optional()

// What it really produces (verified on your Zod 4.4.3):
""        // → 0        ⚠️ a real salary of zero!
null      // → 0        ⚠️
undefined // → undefined ✅ (only this one)
"5000"    // → 5000     ✅
Why .optional() doesn't save you .optional() only lets undefined through untouched. An empty text input submits "", not undefined — so it sails past optional straight into z.coerce.number(), and Number("") is 0. You'd silently store a €0 salary on every blank field.

Your optionalSalaryNumber sidesteps this precisely — a custom before-step that maps every "empty" spelling to undefined first, then validates:

const optionalSalaryNumber = z.preprocess(value => {
  if (value === "" || value === null || value === undefined) {
    return undefined;           // normalise empties BEFORE the gate
  }
  return Number(value);
}, z.number().nonnegative().optional());  // then this validates

// "" → undefined ✅   null → undefined ✅   "5000" → 5000 ✅

z.coerce is a fixed, dumb conversion — great when the built-in Number()/Date() behaviour is exactly what you want. The moment you need a rule about the raw input ("blank means absent"), you've outgrown coerce and want preprocess. You picked correctly.

transform — the tool that runs after

You've already used a transform without naming it: .trim() on z.string().trim() is a built-in transform. It doesn't validate — it edits the value on its way out. A hand-rolled one looks like this:

z.string().transform(s => s.length)   // "hello" → 5
// validates as a string FIRST, then the number falls out the far end

Because transform runs on the far side of the gate, its function receives a value that is already the right type — no defensive typeof checks needed. That's the payoff of "after": you trade the ability to reject for a guarantee of validity.

The type nuance worth planting (ties to your fuzzy-TS spot) Once a schema reshapes data, its input type and output type differ. z.infer<typeof S> gives the output; z.input<typeof S> gives what goes in. For your optionalSalaryNumber: input is unknown, output is number | undefined. That gap is the massaging, made visible in the types.

Where .pipe() fits (you used this too)

Your date fields do z.string().min(1).pipe(z.iso.date(...)). That's neither coerce nor transform — it's two gates in series: first validate "a non-empty string," then feed that string into a second schema that validates "a real ISO date." Reach for pipe when each step is itself a schema, not a plain function. Use preprocess when the first step is a plain function that just cleans input.

Interview-ready sentence "Coercion and preprocessing run before validation to make raw input parseable — coerce for the built-in JS conversions, preprocess when I need a custom rule like 'blank means undefined.' .transform() runs after validation to reshape a value I already know is valid, and it can change the output type — which is why z.input and z.infer can differ."
