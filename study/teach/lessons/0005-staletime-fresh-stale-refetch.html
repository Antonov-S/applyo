staleTime, and the one knob that runs it all

Last lesson your two components shared one request, and navigating away and back didn't refetch — "within the default stale window," I said, and left it there. This is that window. staleTime is the single dial that decides when your cached copy is trusted and when it's chased down again.

In Lesson 1 you built the read path and learned the reframe: you're not fetching, you're caching a copy of server-owned data. That immediately raises the only real question a cache ever asks — when is my copy too old to trust? TanStack Query answers it with one primary setting. Get this one right and you stop fighting the library's "why did it refetch again?!" defaults.

Two states every cache entry lives in

Each query in the cache is either fresh or stale. That's the whole model:

FreshYoung enough to trust. Served from cache only — no network request fires, no matter how many components ask for it.
staleTime
elapses
StaleOld enough to doubt. Still shown instantly from cache, but now a background refetch can fire to refresh it.

staleTime is the age at which fresh becomes stale — measured from when the data last arrived. "As long as data is fresh, it will always come from the cache only. You will not see a network request for fresh data, no matter how often you want to retrieve it" (TkDodo). Stale data isn't wrong or thrown away — it's still rendered — it's merely eligible to be refreshed.

The reframe in one line staleTime doesn't schedule refetches. It sets how long the cache is trusted without checking. Refetches are triggered by events — a mount, a refocus, a reconnect — and only actually fire if the data is already stale when the event happens.

Why your data refetches "for no reason"

Here's the default that surprises everyone: staleTime is 0. Data is stale the instant it arrives. So "every time you e.g. mount a new component instance, you will get a background refetch" (TkDodo). That's not a bug — it's a deliberately safe default (assume data can change any moment), and it's exactly why in Lesson 1 remounting felt so eager.

A stale query auto-refetches on three triggers (Important Defaults):

  • a new component instance mounts (refetchOnMount)
  • the window is refocused — you tab away to a job board and back (refetchOnWindowFocus)
  • the network reconnects (refetchOnReconnect)

With staleTime: 0, every one of those events refetches, because the data is always already stale. Raise staleTime and the same events become no-ops while the clock is running — the trigger fires, sees fresh data, and does nothing. That's the control you were missing.

1. With the default staleTime, when does freshly-arrived data become stale?

Setting the knob

Per-query, in the same useQuery you already wrote — staleTime is in milliseconds:

const { data, isPending, isFetching } = useQuery({
  queryKey: ["opportunities"],
  queryFn: fetchOpportunities,
  staleTime: 20_000,   // trust the cache for 20s; no refetch on mount/focus within it
});

TkDodo's own habit: "I like to set it to a minimum of 20 seconds to deduplicate requests in that time frame, but it's totally up to you." There is "no 'correct' value" — it's a product decision. For Applyo's opportunities list, which you edit rarely and read often, seconds-to-minutes is sensible; for a live-changing feed you'd keep it low. Set it globally as a default when most queries want the same policy:

// in providers.tsx — the QueryClient you were given in Lesson 1
new QueryClient({
  defaultOptions: { queries: { staleTime: 30_000 } }, // every query, unless it overrides
});

The confusion to kill now: staleTime vs gcTime

These get mixed up constantly, and the interview loves the difference. They measure two different clocks:

staleTimegcTime
QuestionIs my copy still trusted?Should I keep this copy at all?
Default0 (stale at once)5 min
Clock startswhen data arriveswhen the query goes unused (no component observing it)
On expirydata is shown but refetch-eligibledata is deleted from the cache

Fresh/stale is about trust; gc is about memory. A query only starts its gcTime countdown once nothing is rendering it — you navigate away and no component observes that key anymore. Come back within 5 minutes and the cached data is still there to show instantly (then a refetch fires if it's stale). Come back after, and it's gone — you're back to a pending first-load. (Note the v5 rename: it was cacheTime in v4.)

2. What does gcTime control, as opposed to staleTime?

What "background refetch" looks like — the L1 payoff

Remember isPending vs isFetching from Lesson 1? Stale-refetching is exactly where that split earns its keep. When a stale query refetches, the old data stays on screen:

  • isPending is false — you have data (the cached copy), so no skeleton.
  • isFetching is true — the queryFn is running in the background.

That's how a good app refreshes silently: the opportunities list never blanks out, it just quietly updates, optionally under a subtle "refreshing…" spinner driven by isFetching. With staleTime: 0 you get this constantly; with a sane staleTime you get it only when the data has genuinely aged. Same machinery, under your control.

Don't confuse the two clocks "Set staleTime high so my data stays cached longer" is the classic mistake. staleTime controls refetching (trust), not retention (memory) — that's gcTime. A query with staleTime: Infinity still gets garbage-collected 5 minutes after nothing renders it.
