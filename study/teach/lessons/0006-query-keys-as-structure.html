Query keys are structure, not labels

So far ["opportunities"] has been a name you picked. Today it becomes an address system — one that decides what shares a cache entry, what refetches when a filter changes, and (next lesson) what a mutation is allowed to invalidate.

You have the read path (Lesson 1) and you control freshness (Lesson 2). Both assumed one query. Real Applyo has many: all opportunities, only the saved ones, only ready_to_apply, one by id. Each is a different slice of server state, and the key is how you tell them apart.

The key does two jobs at once

AddressWhere this data lives in the cache. Same key → same entry → shared data, one request.
one array
DependencyWhat the queryFn reads. Change a value in the key → different entry → automatic fetch.

The docs are explicit about the second job: "query keys act as dependencies for your query functions. Adding dependent variables to your query key will ensure that queries are cached independently, and that any time a variable changes, queries will be refetched automatically" (Query Keys).

The one rule If your queryFn reads a variable, that variable belongs in the queryKey. No exceptions. A filter, an id, a page number, a search term — anything that changes what comes back.

It resembles a useEffect dependency array, but don't stop at the analogy — it's stronger. A changed useEffect dep re-runs the same effect. A changed queryKey points at a different cache entry entirely: the old one isn't overwritten, it stays cached under its own key, so switching back is instant.

How keys are compared

Two rules, and they're not symmetrical (Query Keys):

  • "Query Keys are hashed deterministically!" — "no matter the order of keys in objects". So { status, page } and { page, status } are the same address.
  • "Array item order matters!" — ["opps", "saved", 1] and ["opps", 1, "saved"] are two different addresses.

Hence the convention: fixed segments in array position (generic → specific), variable parameters in a trailing object. Positions carry meaning; object properties don't have to.

1. Which two keys address the same cache entry?

Generic → specific, so invalidation has something to grab

TkDodo's guidance is to "structure your Query Keys from most generic to most specific, with as many levels of granularity as you see fit" (Effective React Query Keys). For Applyo:

["opportunities"]                            // everything opportunity-ish
["opportunities", "list"]                    // all lists
["opportunities", "list", { status: "saved" }] // one filtered list
["opportunities", "detail", id]              // one opportunity

Why the ceremony? Because matching is by prefix. invalidateQueries({ queryKey: ["opportunities"] }) hits every key that starts with it — both ["opportunities"] and ["opportunities", { page: 1 }] are invalidated in the docs' own example (Query Invalidation). Narrow the prefix to ["opportunities", "list"] and you refresh the lists but leave the detail views alone; add exact: true and you hit one entry only.

That's next lesson's machinery — but you have to build the structure today for it to exist tomorrow. Keys designed as flat labels are what make invalidation painful later.

The key factory

Typos in a key string don't error — they silently create a second cache entry, and you debug a "why isn't my list updating" ghost. So: "one Query Key factory per feature", colocated with the queries (TkDodo).

// src/lib/api/opportunities.ts — next to the fetchers that use them
export const opportunityKeys = {
  all:    () => ["opportunities"] as const,
  lists:  () => [...opportunityKeys.all(), "list"] as const,
  list:   (filters: { status?: OpportunityStatus }) =>
            [...opportunityKeys.lists(), filters] as const,
  detail: (id: string) =>
            [...opportunityKeys.all(), "detail", id] as const,
};

Each level builds on the one above, so the hierarchy can't drift. as const keeps the tuple literal — your queryFn can then destructure the key with real types instead of any.

Reading the key inside the queryFn

The queryFn doesn't receive nothing — it gets a QueryFunctionContext, which carries queryKey (plus client, signal, meta). So the key can be the single source of the parameters:

function fetchTodoList({ queryKey }) {
  const [_key, { status, page }] = queryKey
  return new Promise()
}

— straight from the Query Functions docs. Either that, or close over the variable and pass it explicitly — both fine, as long as it's also in the key. Reading it from the context just makes forgetting harder.

What forgetting looks like — and why L2 makes it worse

Here's the interleave. Suppose the filter changes from saved to reviewing, but your key stays ["opportunities"]. The address didn't change, so React Query looks up the same entry — and thanks to your staleTime: 20_000, that entry is fresh. Fresh data is served from cache with no request. The user clicks the filter and nothing happens at all.

With the default staleTime: 0 this bug half-hides: the render triggers a refetch, the new data lands in the wrong address, and it mostly looks right until two filters fight over one entry. A correct staleTime turns a flickery bug into a dead-obvious one. The two settings are one system.

Your win for today

Applyo's opportunities have a real status enum from your Zod schema — saved, reviewing, ready_to_apply, ignored, expired, converted. Today the list gets a status filter, and you watch the cache grow one entry per filter value.

Yours to build & prove

  1. Teach the handler the filter. In src/app/api/opportunities/route.ts, read ?status= off the request URL and filter MOCK; no param = the full list. (Keep it a fixture — five lines, not a backend.)
  2. Filter in the component. useState for the selected status, a <select> of the enum values, and the key becomes ["opportunities", "list", { status }]. Have fetchOpportunities take the status and build the URL.
  3. Watch the cache in Devtools. Flip through three filter values. You should see three separate entries, each with its own fresh/stale clock. Now flip back to one you visited under 20s ago: instant, zero network. That's the address system paying you.
  4. Break it on purpose (30 seconds, do it): drop { status } from the key, keep it in the fetch URL. Predict what the UI does before you click — then click. Restore it. This is the bug you'll now recognise in one guess.
  5. Refactor to a factory. Move the keys into an opportunityKeys object next to the fetchers and use it in the query. Nothing should change at runtime — that's the point.

DoD: it runs, types/lint pass, committed, 2–3 line journal note. Then tell me: why does putting the filter in the key give you instant back-and-forth switching, when a plain useEffect refetch wouldn't? — that answer is the one that shows you own the cache-as-address model.

Interview-ready sentence "A query key is the cache address and the dependency list. Keys are arrays hashed deterministically — object property order doesn't matter, array order does — so I structure them generic to specific, like ['opportunities', 'list', { status }], usually behind a key factory per feature. Anything the queryFn reads goes in the key: change a filter and you're addressing a different cache entry, which refetches automatically while the previous one stays cached for instant switch-back. And because invalidateQueries matches by prefix, that hierarchy is what lets a mutation refresh every list without touching the detail views."
