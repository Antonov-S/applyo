Per-field errors

A failed safeParse hands you a ZodError. Your form needs "put this message under that input." This lesson is the bridge between the two — and it's the direct payoff of the path you already wrote in your salary refines.

You've had the failure branch since Lesson 1: if (!result.success) { result.error }. Today is what you do with result.error — turn one ZodError into messages addressed to individual fields.

The raw material: error.issues

Every ZodError carries a flat array of issues. Each issue has a message and — the key part — a path: an array that is the address of the field that failed. Here's the real output (verified on your Zod 4.4.3) for a bad Opportunity:

// input: title "", sourceUrl "not-a-url",
//        requiredTechnologies ["React", ""], salaryRange { min: 100, max: 50 }
result.error.issues // ⇒
[
  { code: "too_small",      path: ["title"],                    message: "Title is required" },
  { code: "invalid_format", path: ["sourceUrl"],                message: "Invalid URL" },
  { code: "too_small",      path: ["requiredTechnologies", 1],   message: "empty tech" },
  { code: "custom",         path: ["salaryRange", "max"],       message: "min must be ≤ max" },
]
The whole idea in one line path is the field's address. ["salaryRange", "max"] means the max inside salaryRange; ["requiredTechnologies", 1] means the item at index 1. Join it (path.join(".")) and you have a key you can match against an input. That path: ["max"] you wrote in your salary .refine() in Lesson 2? This is what it was for.

1. The 2nd item in requiredTechnologies is empty. What is that issue's path?

Three ways to shape it — matched to the form

You rarely render the raw issues array. Zod v4 gives two reshapers on top of it, and you choose by the shape of your form (docs: Formatting errors):

a. z.flattenError — flat forms, one call

For a form of top-level fields, this is the quick win: a fieldErrors object keyed by field name, each an array of messages.

z.flattenError(result.error) // ⇒
{
  formErrors: [],
  fieldErrors: {
    title:                ["Title is required"],
    sourceUrl:            ["Invalid URL"],
    requiredTechnologies: ["empty tech"],     // ⚠️ which index? lost.
    salaryRange:          ["min must be ≤ max"], // ⚠️ which sub-field? lost.
  }
}
flatten's limit — and why it matters for your form flattenError only keeps the first path segment. Your nested salaryRange.max and your array item requiredTechnologies[1] both collapse onto their top-level key — you can't tell the user which sub-field or which row is wrong. Fine for title/company; not enough for salary or the tech list.

b. z.treeifyError — nested objects & arrays

When the form has nesting (your salaryRange) or lists (your requiredTechnologies), this mirrors the schema's shape so each sub-field keeps its own errors:

z.treeifyError(result.error) // ⇒
{
  properties: {
    title:       { errors: ["Title is required"] },
    requiredTechnologies: {
      errors: [],
      items: [ null, { errors: ["empty tech"] } ]  // null = item 0 is fine
    },
    salaryRange: {
      properties: { max: { errors: ["min must be ≤ max"] } }
    }
  }
}

Read it like the schema: .properties.<field> for object keys, .items[i] for array elements (null where that element passed), .errors for the messages at that node.

2. Your salaryRange.max error must show under the max input. Which helper keeps that detail?

c. Or map issues yourself — the most control

Neither helper is magic; both are built from issues. For a form, a tiny hand-rolled map is often the clearest thing — one message per field, keyed by the joined path:

function toFieldErrors(error) {
  const out = {};
  for (const issue of error.issues) {
    const key = issue.path.join(".");   // "salaryRange.max", "requiredTechnologies.1"
    if (!(key in out)) out[key] = issue.message; // keep the FIRST message per field
  }
  return out; // { title: "Title is required", "salaryRange.max": "min must be ≤ max", ... }
}
Two gotchas worth knowing now 1. A field can have several issues. issues may hold more than one entry for the same path; forms usually show the first, hence the if (!(key in out)) guard. 2. path: [] is a whole-form error. A .refine() with no path attaches to the object root — that's flattenError's formErrors (a form-level banner, not a field message).

Your win for today

Write the bridge once, prove it on your real schema. You implement — I review after:

  1. In a scratch file, import your OpportunitySchema and safeParse a deliberately bad object (empty title, bad URL, min > max, an empty tech string).
  2. Write toFieldErrors(error): Record<string, string> — first message per path.join("."), as above. Type the parameter as z.ZodError.
  3. console.log the result. Confirm you see "salaryRange.max" and "requiredTechnologies.1" as keys — proof the nested/array paths survived.
  4. Then call z.flattenError on the same error and eyeball what got lost. That contrast is the lesson.

Spike rules: scratch file, delete or commit-as-spike after, learning note required. If toFieldErrors feels right, it's a genuine keeper for src/lib/ — we'll wire it into the real form when the form-library lesson lands.
